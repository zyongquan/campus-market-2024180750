## 3. 公共布局组件说明

| 组件名 | 文件位置 | 职责 | 亮点 |
|--------|----------|------|------|
| AppLayout | `src/components/AppLayout.vue` | 整体页面布局 + 页脚 | 包含服务分类/帮助中心/关于我们三栏页脚、版权信息 |
| AppHeader | `src/components/AppHeader.vue` | 顶部操作栏 | 品牌Logo、搜索框（focus蓝边动画）、发布按钮、消息角标(3条未读)、用户头像入口 |
| AppNav | `src/components/AppNav.vue` | 分类导航菜单 | 5个主导航项带emoji图标、`router-link-active` 底部蓝色边框高亮 |

组件层级关系：`App.vue` → `AppLayout` → `AppHeader` → `AppNav` + `RouterView`

**页面骨架**通过 `<RouterView />` 作为动态内容区，所有页面的内容在统一的布局框架内渲染，保证了整个平台视觉风格的一致性。

**路由导航**通过 `<RouterLink>` 组件实现，利用 `.router-link-active` 类名实现当前页面导航项的高亮效果（蓝色文字 + 底部边框）。

**公共布局**的三层嵌套结构确保了：新增页面只需添加路由配置，无需改动布局层；修改公共元素（如Footer）一处改动全局生效。

## 5. 遇到的问题与解决方法

### 问题 1：TypeScript 类型检查失败
**现象**：运行 `vue-tsc --build` 时报两个类型错误 —— `GroupBuyView.vue:75` Tab key 类型不匹配和 `HomeView.vue:106` price 可能为 undefined

**原因分析**：
- Tab key 问题：`tabs` 数组的 `key` 字段被 TypeScript 推断为 `string`，而 `activeTab` ref 的类型是 `'all' | 'group' | 'partner' | 'team'` 联合字面量，`string` 无法赋值给窄类型
- Price 问题：`hotItems` 数组中部分条目（寻物启事、招领信息）没有 `price` 字段，TypeScript 推断为 `number | undefined`

**解决方法**：
- Tab key：在模板中使用类型断言 `tab.key as typeof activeTab`
- Price：使用 `item.price && item.price > 0` 先进行存在性检查，利用 TypeScript 的控制流类型收窄

### 问题 2：AI 初版页面过于简陋
**现象**：第一轮 AI 生成的页面只有标题和一行说明文字（如"这里将展示校园二手商品信息"），看起来不像真实的校园平台

**解决方法**：向 AI 明确要求"将页面做的美化一点，像真实平台"，并提供具体的页面设计要素（搜索栏、筛选Tab、卡片网格、模拟数据、分类切换等）。第二轮生成的结果大幅改善，每个页面都具备了完整的平台级UI。

### 问题 3：构建时的第三方库警告
**现象**：`vite build` 时出现 `[INVALID_ANNOTATION]` 警告（来自 `@vueuse/core`）和 chunk 大小警告

**处理方法**：确认这些警告来自第三方依赖，不影响项目功能。chunk 大小问题可在后续通过代码分割优化，Day2 阶段不需要处理。构建结果：`✓ built in 882ms`，391KB CSS + 1035KB JS。

## 6. 今日反思

**页面骨架**是前端项目的"骨骼系统"。Day1我们理解了项目的业务需求和技术架构，Day2则是将这些理解转化为可视化的页面结构。骨架搭好后，后续的功能填充（Day3-Day7）就有了清晰的承载体。好的页面骨架应该做到：结构清晰（一眼能看出每个区域的功能）、层次分明（布局嵌套合理）、扩展方便（新增页面不影响现有结构）。今天的实践中，我们为8个页面分别设计了符合其业务特点的布局：首页的信息聚合型布局、交易页的搜索筛选型布局、消息页的左右分栏型布局——每个页面的骨架都服务于其核心功能。

**路由导航**是SPA应用的"神经系统"。通过Vue Router的配置，我们建立了URL路径与页面组件之间的映射关系，让用户可以在8个页面之间自由跳转。路由设计的好坏直接影响用户的使用体验——语义化的路径（`/trade` 而不是 `/page2`）让URL本身就有可读性，导航高亮让用户随时知道自己在哪里。我们今天配置的8条路由，路径命名遵循了RESTful风格（资源名词、短横线分隔），为后续可能的权限控制和路由守卫留下了扩展空间。

**公共布局**是团队协作的"契约接口"。通过将Header、Nav、Footer等内容抽取为公共组件，确保了所有页面的视觉一致性，也避免了重复代码。当我们需要修改导航菜单时，只需要改 `AppNav.vue` 一个文件，8个页面全部生效。这种DRY（Don't Repeat Yourself）原则在前端工程化中至关重要。

今天的实践让我深刻理解了：前端开发不是一上来就写复杂的业务逻辑，而是先建立好"骨架"——确定页面结构、路由关系、布局规则和组件边界。这个基础打得越牢固，后续的开发就越顺利。另外，AI在页面美化方面的能力很强——它能快速生成大量模拟数据和样式代码，但开发者需要把控方向（告诉AI要什么风格）、审核质量（检查数据合理性）、修复技术问题（TypeScript类型错误）。人机协作的最佳模式是：人做架构决策和质量把关，AI做批量生成和细节填充。
