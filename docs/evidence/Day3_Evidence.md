## 3. 我的设计

**二手交易（trades）**：校园二手交易的核心是"划不划算"，所以设计了 `price`（售价）和 `originalPrice`（原价）两个字段，让买家一眼看到折扣力度——例如"原价 ¥249，现价 ¥89"。`condition`（成色）在校园场景中很重要，学生买家非常关心物品的新旧程度和使用痕迹（如"有笔记"的教材和"几乎全新"的电子产品完全是两个定位）。`category` 支持按教材书籍、数码电子、生活用品、运动出行等分类浏览，因为不同专业的学生关注点差异很大。`likes` 字段模拟社交互动热度，也为"最受欢迎"排序提供依据。

**失物招领（lostFounds）**：`type` 字段用 `"lost"` 和 `"found"` 区分寻物和招领，这是两种完全不同的用户场景——寻物的人着急找回，招领的人等人来认领。页面上通过 Tab 切换分开展示，标签颜色也不同（橙色/绿色）。`itemName` 与 `title` 分离，`title` 用于列表展示（更吸引眼球），`itemName` 用于精确描述物品。`eventTime` 记录丢失/捡到时间，时间越近找回可能性越大，是用户决策的重要参考。

**拼单搭子（groupBuys）**：`targetCount`（目标人数）和 `currentCount`（当前人数）组成进度系统，如"2/5 人 → 40%"，让用户一眼看到还差几个人，利用紧迫感和从众效应促进参与。`type` 区分三种子场景：拼单（一起买省钱）、搭子（找运动/学习伙伴）、组队（竞赛组队），各自的参与动机不同。`deadline` 给出时间约束，过了截止时间就没有意义了。

**跑腿委托（errands）**：`from`、`to`、`reward` 是跑腿任务的三个核心要素——从哪取、送到哪、给多少钱，缺一不可。页面上 `from` 和 `to` 组成可视化的路线信息（红色起点 → 绿色终点）。`urgency` 区分紧急（🔥 急，红色脉冲动画）和普通任务，让接单人优先处理急件。`taskType` 区分代取快递、代购、打印、排队等类型，不同任务需要的技能和时间不同。

## 4. AI 设计

AI 在本日开发中参与了大部分工作：生成了 `db.json` 的初始数据和结构、四个 API 模块的 TypeScript 类型定义和请求函数、四个列表页面的模板代码、发布页和详情页的框架、Vite 代理配置、以及路由设计。

AI 生成内容的主要问题和我的调整：

1. **发布页只有假提交**：AI 生成的 PublishView 只是设置 `submitted = true` 显示一个成功提示，没有真正调用 API。我完全重写了这个页面：为 5 种发布类型各自设计了对应的表单字段（动态切换），添加了表单验证逻辑，实现了实际的 POST 请求调用，以及 loading 提交状态和成功/失败反馈。

2. **API 模块不完整**：AI 只生成了 `getXxx()` 列表查询函数。我补充了 `getXxxById(id)` 详情查询和 `publishXxx(data)` 发布提交，并且将 id 参数类型从 `number` 扩展为 `number | string`，因为 json-server 会自动生成如 `"BSyxytVAS68"` 格式的字符串 ID。

3. **字段命名不一致**：AI 在页面模板中使用了 `seller`、`campus`、`time` 等字段名，但 `db.json` 和 TypeScript 接口定义中使用的是 `publisher`、`location`、`publishTime`。字段名不匹配会导致页面显示为空。我逐一排查了四个页面中的所有字段引用，统一为 API 接口定义的字段名。

4. **缺少边界状态**：AI 的初始代码只处理了"数据存在就渲染，不存在就显示空状态"两种情况。我补充了 Loading spinner（数据加载中的旋转动画）、Error 提示（显示错误信息 + 重试按钮），形成了完整的四种状态处理流程。

## 6. 遇到的问题与解决方法

### 问题 1：四个页面全部不显示数据（核心问题）

**现象**：二手交易、失物招领、拼单搭子、跑腿委托四个页面打开后全部空白，没有任何数据渲染。

**根因**：前端（Vite 端口 5173）通过 axios 直接向 `http://localhost:3001` 发送跨域请求，且 json-server 未被正确启动。

**解决**：在 `vite.config.ts` 中添加 `server.proxy` 配置，将 `/api` 路径代理转发到 `http://localhost:3001`（自动去掉 `/api` 前缀）；修改 `http.ts` 的 baseURL 从绝对路径 `http://localhost:3001` 改为相对路径 `/api`；启动 json-server。修复后通过 `Invoke-WebRequest` 验证 `/api/trades` 成功返回 10 条数据，所有页面恢复正常。

## 7. 今日反思

Mock 数据建模是前后端分离开发的第一步——通过 `db.json` + `json-server` 模拟出完整的 RESTful API，前端开发者可以在没有真实后端的条件下独立完成从数据定义到页面渲染的全流程。API 接口封装层（`src/api/`）起到了"契约"作用：页面组件不直接依赖后端 URL，而是通过封装的函数获取数据。这意味着后续替换为真实后端时，只需要修改 API 模块内部的实现，所有组件的代码无需变动——这就是分层架构带来的可维护性。

列表渲染的 Loading → Error → Empty → Data 四种状态覆盖是前端用户体验的底线。一个"看起来能用"的页面，背后需要处理网络失败、服务端报错、返回空数据等各种异常情况。这些处理虽然增加了代码量，但让产品从"能跑"变成了"可用"。

Vite 代理机制是今天技术层面最大的收获：开发环境中前端（5173）和后端（3001）是不同端口的不同服务，通过代理统一为一个域，既解决了跨域问题，也让生产环境的路由部署更灵活。这个看似简单的配置，实际上是整个项目能够顺利联调的基石。
